ModelDoctor 实测 · 硬核一图流Higress · lb_policy: prefix_cache

一行负载均衡策略,
把 KV 命中率从 22% 拉到 65%

同一个 Qwen3-32B、同样 4 个 vLLM Pod,只把网关从 cache-blind 改成 KV-aware。不加卡、不换模型、不动引擎参数。
1问题:cache-blind 的网关
默认轮询 / 哈希
相同前缀被打散到 4 个 Pod,每个 Pod 都得把同一段前缀重新 prefill → vLLM APC 几乎命中不了
⇒
结果
前缀缓存命中率只有 22%,算力大量浪费在重复劳动上

OFF/ON 各跑 5 次 × 480 请求取均值。没有 trade-off —— 命中率↑ ⇒ 省 prefill ⇒ 吞吐↑ & 延迟↓。

3机制:Redis 全局前缀树
Higress ai-load-balancer 插件的 prefix_cache 策略 —— 不是 hash 取模,是最长前缀匹配。
① 分块
prompt 切 block
→
② SHA-1
每块取哈希
→
③ 累积 XOR
生成前缀 key
→
④ 最长匹配
Redis 查曾服务的 Pod
→
⑤ pin + APC
命中同一 Pod 的 KV
未命中则回退 global_least_request;failStrategy: FAIL_OPEN —— Redis 挂了放行不阻断(退化为服务自身 LB)。这是插件自带 Redis 实现,不是 GIE/EPP(llm-d)那条路径。

引擎上报的 block 查询/命中:ON 让每个 Pod 少做了几百万个 block 的重复 prefill。这才是命中率 22%→65% 底下真正发生的事。
5关键:靠调度,不靠堆负载
✓ 命中率均匀抬高
4 个 Pod 的 ON 命中率全部落在 58–70%,没有谁被冷落。
✓ 流量没有热点
最大 Pod 份额 OFF/ON 都在 27–29%,开关前后几乎不变。增益纯来自调度。
6会踩的坑
!
只在共享前缀负载上赢。短/随机前缀几乎无收益,还要付 Redis 查询 + 回退开销。
!
Redis 是全局状态单点。redisKeyTTL 要按前缀热度调;FAIL_OPEN 下 Redis 抖动会悄悄吃命中率。
!
块粒度要对齐。插件 block 级 SHA-1 与 vLLM KV block 粒度不匹配会压低命中(Mooncake 教训)。
!
小规模结论。本测仅 4 Pod;扩容后前缀树命中分布、Redis 压力都会变,需重压测。
一句话:prefix_cache = 把「请求该去哪个 Pod」从 cache-blind 改成 KV-aware。在对的负载上,这几乎是一顿免费的午餐。
