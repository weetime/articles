# 1. Capacity sweep: plot the latency-throughput curve, find the goodput knee (--warmup drops cold start) $ guidellm benchmark run --target $URL --model $MODEL \ --data 'prompt_tokens=512,output_tokens=128' --warmup 0.2 \ --rate-type concurrent --rate 1,2,4,8,16,32,64,128,192 --max-seconds 40# 2. Set an SLO (e.g. P95 ≤ 2s); goodput = max throughput within the SLO# 3. Validate within goodput, under realistic arrivals (run poisson, not just constant) $ guidellm ... --rate-type constant --rate 12 # steady rate $ guidellm ... --rate-type poisson --rate 12 # bursty arrivals (conservative goodput)
rate-type · which to use for goodput
synchronous
one at a time → latency floor · single-stream baseline (best case for SLO)
concurrent/sweep
sweep capacity → plot the curve, locate the goodput knee
constant
fixed QPS, steady → goodput under steady-state operation
poisson
bursty (Poisson) arrivals → conservative goodput (tail more likely to breach SLO)
throughput
max concurrency → raw peak / hardware ceiling (usually in violation; not deliverable capacity)
