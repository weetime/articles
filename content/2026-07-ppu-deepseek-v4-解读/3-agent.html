02 · Agent 场景真实工具调用 trace 回放

换成长输入的 Agent 负载,单实例扛不住了

同一台机器、同样默认配置,把负载从短对话换成真实 Agent 轨迹(长输入 ~5–9K token + 短输出)。表现完全变样。

首字延迟 TTFT(中位数)

65–108 秒

用户点一下,要等一分多钟才见第一个字。短对话时这个数只有零点几秒。

实际处理速度

0.6–1.7 req/s

而轨迹本身的到达速度是 3.9 req/s——供不应求 2–7 倍,请求全堆在队列里。

为什么? Agent 的特征是长输入(预填充很重)+ 短输出(几十个 token 就结束),瓶颈卡在算力,不在并发数、也不在显存。所以上一张图那套"调并发"的办法,在这里不管用——把上下文放长到 64K 反而更慢(放进来的长请求预填充更重)。
